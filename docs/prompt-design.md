# プロンプト設計

aiwrite のプロンプトはすべてサーバーサイドの **TypeScript ソースコード** として管理されています。本ドキュメントでは、その設計思想・構造・ベストプラクティスを体系的に記述します。

---

## 1. 設計思想

### 1.1 コンテキストエンジニアリングとしてのプロンプト管理

2026年現在、プロンプト設計は単なる「テキストの書き方」ではなく、**コンテキストエンジニアリング**（LLM のコンテキストウィンドウに何を・どの順序で・どの粒度で載せるかを設計する行為）として捉えられています。aiwrite はこの考え方に基づき、プロンプトを **構造化されたロジック** としてコード内に統一管理しています。

### 1.2 コード管理を選択する根拠

aiwrite のプロンプト構築は、単純なテンプレート変数埋め込みではなく、以下のようなプログラム的な振る舞いを含みます。

- **メッセージ配列の構造的組み立て**: `system` → 過去ログ（直近10件を `slice(-10)` で切り出し）→ 最終ユーザーメッセージ、という複数メッセージの配列を構築する
- **フェーズ × モードの組み合わせ**: `phaseGuides`（4種）と `instructionByMode`（3種）を実行時に選択・結合する
- **2段階API呼び出しの連携**: メイン会話のあとにボード更新プロンプトを呼び、その結果をフロント側でマージする

これらの構造的ロジックはテンプレートエンジンや Prompt CMS（LangSmith, Langfuse 等）のテンプレート機能では表現しきれず、結局コード側に残ります。「テンプレートだけ外部化しても、ロジックがコードに残るなら二重管理になる」ため、コード内に統一しています。

### 1.3 補助的な判断根拠

| 観点 | 判断根拠 |
|---|---|
| **型安全性** | TypeScript で変数の抜け漏れをコンパイル時に検知できる |
| **開発者 = プロンプト設計者** | 非エンジニアがプロンプトを編集する運用がないため、GUI編集のメリットが不要 |
| **バージョン管理** | Git で差分・履歴が自然に追跡される |
| **集約度** | プロンプト構築ロジックは `lib/openrouter.ts` と `lib/board-update.ts` の2ファイルに集約済み |

### 1.4 CMS移行を検討すべきタイミング

以下の条件が複数揃った場合に、Prompt CMS（Portkey, Langfuse 等）への移行を検討します。

- チームが増え、非エンジニアがプロンプトを直接編集する必要が出る
- プロンプト品質のA/Bテストがビジネス上重要になる
- ユーザー数増加に伴い、品質劣化の自動検知が必要になる

## 3. 統合アーキテクチャ：メッセージパイプライン

aiwrite は毎回のAPI呼び出しで、**作家性（ペルソナ）**・**ボード（作品設定）**・**会話履歴**の3要素を1つのリクエストに統合してAIと対話します。

### 3.1 リクエストの全体構造

```
┌──────────────────────────────────────────────────────────────┐
│ [system] システムプロンプト                                     │
│                                                              │
│   ① 静的コンテンツ（キャッシュ効率の高い順）                      │
│   ├─ AIの基本役割（小説に伴走するAI作家）                        │
│   ├─ フェーズ別ガイド（構造化/試し書き/微調整/本稿）               │
│   │                                                          │
│   ② 半動的コンテンツ（セッション単位で変化）                      │
│   ├─ 作家ペルソナ（名前・文体・視点・得意ジャンル）           ← 作家性│
│   └─ ボード全内容（タイトル・コンセプト・世界観・              ← ボード│
│       プロット・キャラクター・参照メモ・文体ルール）               │
├──────────────────────────────────────────────────────────────┤
│ [user/assistant × N] 直近10件の会話ログ                   ← 会話履歴│
├──────────────────────────────────────────────────────────────┤
│ [user] 最終指示メッセージ                                       │
│                                                              │
│   ③ 動的コンテンツ（毎回変化）                                  │
│   ├─ フェーズ別指示 + モード別指示                               │
│   ├─ 現在の原稿全文                                            │
│   └─ ユーザーが今入力したテキスト                                │
└──────────────────────────────────────────────────────────────┘
```

ボードの内容と作家ペルソナは **system プロンプトに毎回全文埋め込み** されます。ユーザーがボードを手動で編集した場合も、次のAPI呼び出しで即座に反映されます。

### 3.2 コンテンツ配置の設計原則

プロンプト内のコンテンツ配置は **「静的コンテンツを先頭に、動的コンテンツを末尾に」** という原則に従っています。

| 層 | 内容 | 変化頻度 | 配置 |
|---|---|---|---|
| 静的 | AIの基本役割・制約 | 変わらない | system 冒頭 |
| 半動的 | フェーズガイド・ペルソナ・ボード | 作品/フェーズ切替時 | system 後半 |
| 動的 | 会話履歴 | 毎ターン | messages 中間 |
| 動的 | 原稿全文・ユーザー入力 | 毎ターン | 最終 user メッセージ |

この配置により、将来プロンプトキャッシュ（OpenAI Prompt Caching / Anthropic Cache Control 等）を導入した際に、system プロンプトの先頭部分がキャッシュヒットしやすくなります。

### 3.3 記憶の外部化：ボード自動更新ループ

会話履歴は直近10件に制限されているため、長い議論の文脈は時間とともに失われます。この制限を補っているのが、**ボードの自動更新による記憶の外部化（Semantic Memory パターン）** です。

```
ユーザーの指示
     ↓
/api/openrouter → AIの回答を取得
     ↓
/api/board-update → 会話内容からキャラ・世界観・プロット等を自動抽出
     ↓
/api/ボード（settings）を更新
     ↓
次の会話で system プロンプトに最新ボードが埋め込まれる ← ここで記憶が循環する
```

このループにより、10件前の会話で決まったキャラ設定や世界観は、ボードに「構造化された記憶」として残り、system プロンプト経由で以降のすべての会話に影響を与え続けます。会話ログの切り捨てとボードの永続化を組み合わせることで、**有限のコンテキストウィンドウ内で長期的な創作文脈を維持**しています。

| 記憶の種別 | aiwrite での実装 | 説明 |
|---|---|---|
| **Semantic Memory** | 創作ボード（NovelSettings） | キャラ・世界観・プロット等の構造化された知識 |
| **Episodic Memory** | 直近10件のチャットログ | 最近の対話のコンテキスト |
| **Procedural Memory** | フェーズガイド・モード指示 | 各フェーズでの振る舞いルール |

### 3.4 原稿の記憶範囲と制約

AIには最終ユーザーメッセージの一部として **現在の話の原稿全文** が毎回渡されます（`currentManuscript: activeEpisode.manuscript`）。しかし、**過去の話の原稿は含まれません**。

| 情報 | AIに渡されるか |
|---|---|
| 現在の話の原稿全文 | ✅ 毎回送信 |
| 過去の話の原稿（第1話→第2話→…） | ❌ 送信されない |
| ボードのプロット欄 | ✅ system プロンプトに含まれる |

つまり第3話を書いているとき、AIは第3話の本文は把握していますが、第1話・第2話で具体的に何が起きたかは知りません。ボードの「プロット」欄にあらすじが書いてあれば推測できますが、本文レベルの詳細（伏線の表現、キャラの台詞回し、死亡・退場の描写など）は失われます。

**長編で問題になるケース:**

- 前の話で張った伏線を回収する描写を書けない
- 退場済みのキャラを再登場させてしまう
- 文体や口調が話ごとにブレる

**将来の対策案:**

1. **「既刊あらすじ」フィールドの追加** — 各話の完了時にAIに要約を自動生成させ、ボードに蓄積する。system プロンプトで過去の話の文脈を伝えつつトークンを節約できる
2. **直前の話の末尾を含める** — 話の接続部分（前話のラスト数百文字）を `currentManuscript` に先頭付与することで、文脈の滑らかな接続を担保する

## 4. メイン会話プロンプト（`lib/openrouter.ts`）

ユーザーがテキストを送信するたびに呼ばれる、AIとの主たる会話を担うプロンプトです。

### 4.1 システムプロンプト（`buildSystemPrompt`）

以下を結合して1つの `system` メッセージを構成します。

```
[基本役割プロンプト]
[フェーズ別ガイド]

AIのキャラクター・トーク: {persona.character}
ユーザーの呼び方: {persona.userCall}
作家名: {persona.name}
執筆する文体: {persona.style}
執筆する視点: {persona.pointOfView}
執筆の得意ジャンル: {persona.genres}

作品タイトル: {settings.title}
コンセプト: {settings.concept}
世界観: {settings.worldView}
プロット: {settings.plot}
キャラクター設定: {settings.characters}
参照リンクやメモ: {settings.referenceLinks}
文体・本稿ルール: {settings.writingRules}
```

#### 4.1.1 プロンプトの6要素マッピング

システムプロンプトは、プロンプト設計のベストプラクティスで推奨される6つの要素（Six Pillars）を網羅しています。

| 要素 | aiwrite での対応 | 出典（`lib/prompts.ts`） |
|---|---|---|
| **Role/Persona** | AI作家の基本定義 + 個別ペルソナ | `SYSTEM_PROMPT_HEADER` + `DEFAULT_PERSONA` |
| **Goal/Task** | フェーズ別ガイド | `PHASE_METADATA[].guide` |
| **Context/References** | ボード全内容（世界観・キャラ・プロット等） | `NovelSettings` データ |
| **Format/Output** | フェーズ別指示 + モード別指示 | `PHASE_METADATA[].instruction` + `INSTRUCTIONS_BY_MODE` |
| **Examples** | （現状未実装） | — |
| **Constraints** | 「ユーザーの主体性を尊重し…」等の振る舞い制約 | `SYSTEM_PROMPT_HEADER` |

#### 4.1.2 フェーズ別定義の詳細

フェーズごとの役割や指示は `PHASE_METADATA` で定義されています。

| フェーズ ID | AIの役割（UI） | ガイドの要旨（`guide`） |
|---|---|---|
| `structuring` | 編集者兼リサーチャー | テーマをコンセプト・勢力図・キャラ・プロットへ変換 |
| `sampling` | サンプル脚本家 | 短い本文サンプルを出し、読み味やトーンを検証 |
| `refining` | 共著者兼編集者 | ユーザーの反応を反映し、本稿用設定を固定 |
| `drafting` | ゴーストライター | 固定済み設定から章・話単位の本文を生成 |

### 4.2 ユーザーメッセージの構成

最終的にAPI（OpenRouter）へ送信されるユーザーメッセージは、以下のパーツを結合して生成されます。

```
[フェーズ別指示（PHASE_METADATA[].instruction）]
[モード別指示（INSTRUCTIONS_BY_MODE）]

現在の原稿:
{現在のエピソードの本文}

ユーザー依頼:
{入力フォームのテキスト}
```

### 4.3 会話履歴の扱い

- 直近 **10件** のメッセージのみ OpenRouter に送信（`messages.slice(-10)`）
- それ以前のログは `localStorage` に保存されますが、APIには含まれません
- 会話履歴が消えても、**ボード自動更新** により重要な設定は `system` プロンプトへ永続化されます。

## 5. ボード自動更新プロンプト（`lib/board-update.ts`）

ユーザーとAIの会話が終わるたびに、その内容から創作ボードを自動更新するために呼ばれるプロンプトです。

### 5.1 動作フロー

```
1. ユーザーが送信
2. /api/openrouter → AIの本文回答を取得
3. /api/board-update → 同じ会話をもとにボード設定をJSON抽出
4. フロント側でJSONをマージ（空文字・未変更の項目は既存値を維持）
```

### 5.2 システムプロンプト

`BOARD_UPDATE_PROMPTS.SYSTEM` 定数により管理されています。

**設計上のポイント:**

- **明確な出力契約**: JSON スキーマを明示し、出力形式のブレを防ぐ
- **保守的な更新ルール**: 「却下された案」「一時表現」を固定設定にしない指示で過剰反映を抑制
- **既存設定の保全**: 「既存設定を破壊せず」の指示で、ボードの蓄積的な成長を保証

## 6. ベストプラクティスと改善指針

### 6.1 Few-Shot Examples の活用

**現状**: フェーズ別ガイドは抽象的な指示文のみで、出力の具体例（Few-Shot）は含まれていません。
**推奨**: 特に `drafting` フェーズでは、期待する文体・構成を示す短い例文を `lib/prompts.ts` に追加することを検討してください。

### 6.2 プロンプトキャッシュへの準備

system プロンプトの先頭部分（`SYSTEM_PROMPT_HEADER`）を固定することで、API側のキャッシュ機能を最大限活用できます。

### 6.3 セルフレビュー指示（Self-Evaluation）

AIに出力前の自己チェックを促す指示を追加することで、キャラの口調ブレを低減できます。

## 7. プロンプトのカスタマイズ箇所ガイド

aiwrite のプロンプトの内容（定数）はすべて **`lib/prompts.ts`** に集約されています。目的に応じて、このファイル内の各定数を編集して調整します。

### 7.1 定数別のカスタマイズ対象

| 変更したい内容・目的 | 編集対象（`lib/prompts.ts` 内の定数） |
|---|---|
| **AI作家のデフォルト設定**（名前、初期文体、呼び方など） | `DEFAULT_PERSONA` |
| **全フェーズ共通の根本的な役割定義**（「あなたはAI作家です...」など） | `SYSTEM_PROMPT_HEADER` |
| **各フェーズでの立ち回り・指示文**（構造化/試し書き/微調整/本稿） | `PHASE_METADATA` の各要素の `guide` および `instruction` |
| **UI上の表示内容**（フェーズ名、プレースホルダ、入力例） | `PHASE_METADATA` の各要素の `label`, `placeholder`, `starters` |
| **チャット/本文/リテイク別の基本指示** | `INSTRUCTIONS_BY_MODE` |
| **ボード自動更新（記憶抽出）のルール** | `BOARD_UPDATE_PROMPTS` |

### 7.2 APIパラメータ（出力のブレ幅など）

APIの挙動自体を調整する場合は、以下のファイルを編集します。

| 変更したい内容・目的 | 編集対象ファイルと箇所 |
|---|---|
| **生成の創造性（ランダム性）**を調整したい | `app/api/openrouter/route.ts` の `temperature` |
| **設定抽出の確実性**を調整したい | `app/api/board-update/route.ts` の `temperature` |

### 変更時のチェックリスト

プロンプトの内容を変更する際は、`lib/prompts.ts` を編集したあと、以下を確認してください。

- [ ] 変更内容が TypeScript の型定義（`lib/types.ts`）に準拠しているか
- [ ] 変更前後で同じユーザー入力に対する出力を比較したか（リグレッションテスト）
- [ ] 変更した定数がフロントエンドのUI（作品設定など）に正しく反映されているか
- [ ] ボード更新プロンプトとメイン会話プロンプトの整合性が取れているか
